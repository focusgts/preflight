# ADR-005: Weighted Scoring Algorithm for Migration Readiness Assessment

## Status: Accepted

## Date: 2026-03-21

## Context

The assessment phase produces a readiness score that drives customer-facing decisions: whether to proceed, what to fix first, and how much budget to allocate. The score must be transparent (customers must understand why they got a specific number), comparable (scores across different migration types should be meaningful), and actionable (each sub-score should map to specific remediation steps).

Different aspects of migration readiness contribute unequally to overall risk. Code compatibility issues are harder and more expensive to fix than content quality issues. Integration complexity compounds non-linearly as the number of dependencies grows.

## Decision

We use a weighted scoring algorithm with five categories, each scored 0-100 and combined using fixed weights:

### Category Weights

| Category | Weight | Rationale |
|----------|--------|-----------|
| Code Compatibility | 0.30 | Largest driver of migration effort and risk. Blockers here halt the entire migration. |
| Content Readiness | 0.20 | Content volume and quality affect timeline. Broken references and duplicates cause post-migration issues. |
| Integration Complexity | 0.20 | Third-party integrations are the most common source of migration failures. |
| Configuration Readiness | 0.15 | Configuration migration is usually automatable but can cause subtle production issues. |
| Compliance | 0.15 | Compliance requirements add constraints but rarely block migration entirely. |

### Overall Score Formula

```
overallScore = round(
  codeCompatibilityScore * 0.30 +
  contentReadinessScore * 0.20 +
  integrationComplexityScore * 0.20 +
  configurationReadinessScore * 0.15 +
  complianceScore * 0.15
)
```

### Per-Category Scoring

Each category score is computed from individual findings:

```
categoryScore = 100 - sum(findingPenalties)

findingPenalty = basePenalty(severity) * compatibilityMultiplier
```

| Severity | Base Penalty |
|----------|-------------|
| Critical | 25 points |
| High | 15 points |
| Medium | 8 points |
| Low | 3 points |
| Info | 0 points |

| Compatibility Level | Multiplier |
|--------------------|-----------|
| Blocker | 1.5x |
| Manual Fix | 1.0x |
| Auto-Fixable | 0.5x |
| Compatible | 0.0x |

The score is clamped to [0, 100]. A single CRITICAL + BLOCKER finding applies a 37.5-point penalty, ensuring that any blocker dramatically reduces the category score.

### Readiness Levels

| Score Range | Level | Meaning |
|-------------|-------|---------|
| 80-100 | Ready | Migration can proceed with minimal additional work |
| 60-79 | Ready with Work | Specific issues must be addressed but migration is viable |
| 40-59 | Significant Work Needed | Major remediation required before migration |
| 0-39 | Not Recommended | Fundamental blockers exist; consider alternative approaches |

### Confidence Level

Each assessment also produces a confidence level (0.0-1.0) based on:
- Completeness of source environment scan (did we scan all code, content, and config?)
- Number of findings relative to environment size (too few findings may indicate incomplete scan)
- Whether all integration endpoints were reachable during assessment

## Consequences

**Positive:**
- Transparent: customers can see exactly which findings reduced their score and by how much
- Actionable: fixing a specific finding has a calculable impact on the overall score
- Comparable: a score of 72 means roughly the same thing across different migration types
- Auto-fixable findings are penalised at half rate, incentivising use of automated remediation
- Blocker multiplier ensures critical issues cannot be hidden by high scores in other categories

**Negative:**
- Fixed weights may not perfectly reflect reality for all migration types (AEM code modernisation vs. analytics data migration have very different weight profiles)
- Score can be gamed by resolving many low-severity findings while ignoring fewer high-severity ones
- Penalty-based scoring means an environment with many findings but all auto-fixable may score lower than expected

**Mitigations:**
- Per-migration-type weight overrides planned for v2 (e.g., analytics migrations may weight integration higher)
- Dashboard highlights blocker count separately from overall score
- Assessment report includes both the numeric score and a narrative summary generated by AI

## Alternatives Considered

**Simple average:** Equal weighting across categories. Does not reflect the reality that code compatibility issues are far more impactful than content quality issues.

**Machine-learned weights:** Train weights on historical migration outcomes. Requires a dataset of completed migrations we do not yet have. Planned for v2 once we have 50+ completed migrations with outcome data.

**Pass/fail with no numeric score:** Simpler but provides no granularity. Customers need to understand the degree of readiness to plan budgets and timelines.

**Unweighted finding count:** Counts findings by severity without combining into a score. Loses the ability to compare across environments of different sizes.
