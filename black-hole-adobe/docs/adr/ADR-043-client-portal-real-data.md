# ADR-043: Client Portal & Real Data Integration

## Status: Proposed

## Date: 2026-03-30

## Context

The client portal at `src/app/(portal)/portal/[orgId]/page.tsx` uses `getMockData(orgId)` for all data — phase progress, stats, savings comparison, knowledge base. It has no authentication — anyone with the URL can access it. The portal is the customer-facing view of their migration and must show real data to be credible.

A customer checking their migration status and seeing obviously fake progress bars and savings numbers will question every other number in the platform. The portal must reflect reality.

## Decision

### 1. Wire portal to real migration data

- Replace `getMockData(orgId)` with fetch to `GET /api/migrations?organizationId={orgId}`
- Phase timeline: derive from real migration phases and their statuses
- Stats cards: compute from real migration metrics (items migrated, time saved, issues resolved)
- Savings comparison: use real `CostEngine` estimate vs actual progress

### 2. Add portal authentication

- Create shareable portal links with org-scoped access tokens: `/portal/{orgId}?token={shareToken}`
- Tokens are generated by admin when creating a migration, stored in SQLite
- Tokens can be revoked, have expiry (30 days default)
- No Adobe/Black Hole account required for client — just the link
- Add `POST /api/portal/tokens` to generate tokens (admin-only)
- Add `DELETE /api/portal/tokens/[id]` to revoke tokens
- Add middleware in `src/app/(portal)/portal/[orgId]/layout.tsx` that validates token before rendering

### 3. Wire knowledge base

- KB articles generated from migration findings and remediation guidance
- Store as migration-scoped records in SQLite
- Display in portal with search and category filtering
- Add `GET /api/portal/{orgId}/knowledge-base` endpoint

### 4. Wire client portal layout (`src/app/(portal)/portal/[orgId]/layout.tsx`)

- Add portal-specific header with organization name
- Add navigation between overview, migration detail, and knowledge base
- Add "Contact Us" and "Request Assessment" CTAs

## Consequences

- Positive: Clients see their real migration status — builds trust and reduces support calls
- Positive: Shareable links mean clients don't need accounts — zero friction
- Positive: Knowledge base provides self-service answers
- Negative: Portal auth via shareable links is less secure than full auth — acceptable for migration status visibility
- Negative: Portal data is only as good as the underlying migration data

Estimated effort: 14-18 hours
